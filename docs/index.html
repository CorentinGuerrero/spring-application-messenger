Spring-native application messaging

One API for commands, queries, events, and progressive async routing.

A lightweight Spring Boot library inspired by Symfony Messenger: start in-process, add middleware, route selected messages to RabbitMQ, Kafka, Redis Streams, or a JDBC outbox, and keep application handlers readable.

Architecture

Application code talks to buses. The library wraps messages in envelopes, applies middleware, resolves routing, then invokes a local handler or hands the message to a transport.

Application Dispatches a command, asks a query, or publishes an event.
Envelope Adds metadata: message ID, correlation ID, causation ID, timestamp, headers.
Middleware Validation, transactions, logging, authorization, metrics, retries.
Router Chooses sync, RabbitMQ, Kafka, Redis, or JDBC outbox.
Handler Incoming messages are converted back and dispatched to application handlers.

Messages And Buses

The public API deliberately stays small. Commands modify state, queries read state, and events describe facts that already happened.

CommandBus

Commands

One command has exactly one handler. It can return a result and failures are propagated to the caller.

public record CreateOrder(...)
    implements Command<OrderId> {
}
QueryBus

Queries

One query has exactly one handler. It always returns a result and remains synchronous by design.

public record GetOrder(...)
    implements Query<OrderView> {
}
EventBus

Events

An event may have zero, one, or many handlers. Handlers do not return values.

public record OrderCreated(...)
    implements Event {
}

Handlers

Handlers are regular Spring beans. At startup, the scanner validates annotated beans, finds handle(...) methods, infers message types, and fails fast on invalid configuration. 

@CommandHandler
public class RegisterUserHandler {
    private final UserRepository repository;
    private final EventBus eventBus;

    public UserId handle(RegisterUser command) {
        User user = User.register(command.email());
        repository.save(user);
        eventBus.publish(new UserRegistered(user.id(), user.email()));
        return user.id();
    }
}

@EventHandler
public class SendWelcomeEmailHandler {
    public void handle(UserRegistered event) {
        // send email
    }
}

Validation rules

  • Handlers expose one or more handle(...) methods.
  • Each handler method has exactly one message parameter.
  • Commands and queries have one handler.
  • Events can have many handlers.
  • Command/query return types must match the message contract.
  • Event handlers must return void.

Middleware

Middleware runs around dispatch before the final local handler or selected transport. It is the extension point for cross-cutting behavior.

Typical middleware

  • Bean Validation
  • Transaction boundaries
  • Logging and correlation IDs
  • Authorization
  • Metrics and tracing
  • Local retry or error mapping

API

public interface MessageMiddleware {
    Object invoke(
        MessageEnvelope envelope,
        MessageHandler next
    );
}

Routing Configuration

Routing decides where a command or event goes. Queries are always synchronous and are not routed to brokers. 

messenger:
  routing:
    commands:
      RegisterUser: sync
      SendWelcomeEmail: rabbitmq
      RebuildUserSearchIndex: kafka
    events:
      UserRegistered: pg-outbox
Routing supports simple class names such as UserRegistered and fully qualified class names. Use fully qualified names when two messages share the same simple name.

Transports

Transport modules are optional. Add only the adapters your application needs.

Transport Name Module Purpose
In-process sync messenger-core Default local dispatch for handlers. Queries always use this path.
RabbitMQ rabbitmq messenger-transport-rabbitmq Publishes TransportMessage to exchanges and consumes queues back into handlers.
Kafka / Redpanda kafka messenger-transport-kafka Uses Spring Kafka templates, listeners, consumer groups, topics, and headers.
Redis Streams redis messenger-transport-redis Stream publishing with managed consumer group listener and ACK support.
JDBC outbox pg-outbox messenger-transport-jdbc-outbox Stores messages in messenger_outbox inside the database transaction.

Transport Message Format

Broker adapters move a TransportMessage. Consumers convert the payload back to the declared Java type and dispatch it to the matching local handler. 

{
  "messageType": "com.acme.UserRegistered",
  "busType": "EVENT",
  "transportName": "kafka",
  "payload": {
    "userId": "4b7b...",
    "email": "john@example.com"
  },
  "metadata": {
    "messageId": "...",
    "correlationId": "...",
    "causationId": null,
    "createdAt": "...",
    "headers": {}
  }
}

Headers

  • messenger_message_id
  • messenger_correlation_id
  • messenger_causation_id
  • messenger_bus_type
  • messenger_message_type

JDBC Outbox

The outbox stores messages in the same database transaction as business changes. A publisher then forwards pending rows to the configured target transport.

Command handler Saves business state and publishes an event.
Outbox transport Writes the event to messenger_outbox.
Commit Business row and outbox row succeed or fail together.
Publisher Reads pending rows, forwards them, and marks them published or failed. 
messenger:
  transports:
    jdbc-outbox:
      enabled: true
      initialize-schema: true
      table-name: messenger_outbox
      publisher:
        enabled: true
        batch-size: 50
        max-attempts: 3
        target-transport: kafka

Complete Spring Boot Example

The repository includes spring-messenger-example, a runnable app with one global configuration: sync command/query, JDBC outbox, RabbitMQ command, and Kafka/Redpanda command.

Run the app

The default endpoints are localhost:5672 for RabbitMQ and localhost:9092 for Kafka/Redpanda.

./gradlew :spring-messenger-example:bootRun

Run broker tests

Testcontainers starts PostgreSQL, RabbitMQ, and Redpanda to validate real message paths.

./gradlew :spring-messenger-example:test --tests '*ContainerTest'

Testing Helpers

messenger-test is designed for application tests. It is different from the sample app: it gives reusable test primitives.

Fake buses

Use FakeCommandBus and FakeQueryBus to capture dispatched messages and configure return values.

Event recorder

Use RecordingEventBus to verify events published by a handler without starting Spring.

Assertions

Use MessengerAssertions.assertThat(...) for fluent event assertions.

RecordingEventBus eventBus = new RecordingEventBus();
RegisterUserHandler handler = new RegisterUserHandler(repository, eventBus);

handler.handle(new RegisterUser("john@example.com", "John"));

assertThat(eventBus)
    .hasPublished(UserRegistered.class)
    .hasPublishedSatisfying(UserRegistered.class, event ->
        assertEquals("john@example.com", event.email())
    );

Status And Scope

The current project is a 0.1.0-SNAPSHOT. The core programming model is in place, but production hardening is still ongoing.

Implemented

  • CommandBus, QueryBus, EventBus
  • Spring annotations and handler scanning
  • Middleware chain and Bean Validation
  • Routing and transport SPI
  • RabbitMQ, Kafka, Redis, and JDBC outbox adapters
  • Test helpers and Testcontainers integration tests

Not v1 yet

  • DLQ handling
  • Serialization strategy customization
  • Micrometer and tracing integration
  • Async local executor dispatch
  • Schema alias/version contract for non-Java interoperability
