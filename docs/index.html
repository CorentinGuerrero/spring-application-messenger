Lightweight Spring Boot messaging

Commands, queries, and events without the framework gravity.

Spring Application Messenger gives Spring applications a simple bus API, automatic handler discovery, middleware, synchronous dispatch by default, and optional routing to RabbitMQ, Kafka, Redis Streams, or a JDBC/PostgreSQL outbox.

Quickstart

The programming model is intentionally small: define a message, annotate a Spring handler, then dispatch through the matching bus.

Command

Change application state

Exactly one command handler. May return a result. Exceptions are propagated to the caller.

Query

Read application state

Exactly one query handler. Always synchronous and in-process by design.

Event

React to something that happened

Zero, one, or many event handlers. No return value. Error strategy is configurable.

public record RegisterUser(String email) implements Command<UserId> {
}

@CommandHandler
public class RegisterUserHandler {
    private final UserRepository repository;
    private final EventBus eventBus;

    public UserId handle(RegisterUser command) {
        User user = User.register(command.email());
        repository.save(user);
        eventBus.publish(new UserRegistered(user.id(), user.email()));
        return user.id();
    }
}

UserId userId = commandBus.dispatch(new RegisterUser("john@example.com"));

Runtime Model

Every dispatch follows the same shape. Middleware can add validation, transactions, observability, authorization, retries, correlation IDs, or custom behavior before the final handler or selected transport.

1. Bus Application code calls CommandBus, QueryBus, or EventBus.
2. Envelope The message receives metadata: message ID, correlation ID, causation ID, headers.
3. Middleware Validation, transactions, logging, and custom policies run around dispatch.
4. Handler or transport The router chooses sync, RabbitMQ, Kafka, Redis Streams, or JDBC outbox.

Routing

Application code does not change when a message moves from local sync handling to a broker. Routing is configuration. 

messenger:
  routing:
    commands:
      RegisterUser: sync
      SendWelcomeEmail: rabbitmq
      RebuildUserSearchIndex: kafka
    events:
      UserRegistered: pg-outbox
  transports:
    rabbitmq:
      exchange: messenger.example
      routing-keys:
        SendWelcomeEmail: mail.welcome
      consumer:
        enabled: true
        queues: messenger.example.mail
    kafka:
      topics:
        RebuildUserSearchIndex: messenger.example.user-index
      consumer:
        enabled: true
        topics: messenger.example.user-index
        group-id: messenger-example
    jdbc-outbox:
      initialize-schema: true
      publisher:
        target-transport: sync

Transports

The current transport layer covers local sync dispatch, RabbitMQ, Kafka-compatible brokers, Redis Streams, and a JDBC outbox. Queries stay synchronous.

Transport Name Use case Notes
In-process sync Local command, query, and event handlers Default behavior. Queries always use this path.
RabbitMQ rabbitmq Queue-backed async commands or events Uses Spring AMQP and dispatches consumed messages back to application handlers.
Kafka / Redpanda kafka Topic-backed async commands or events Uses Spring Kafka configuration for broker security and client options.
Redis Streams redis Stream-backed local async workflows Includes managed listener support with consumer group ACK.
JDBC outbox pg-outbox Transactional event persistence before broker publishing Creates and uses a stable messenger_outbox table structure.

Complete Example

The repository includes a Spring Boot example app with one global configuration: sync command/query, PostgreSQL-style outbox, RabbitMQ async command, and Kafka/Redpanda async command.

Run the app

The default endpoints are localhost:5672 for RabbitMQ and localhost:9092 for Kafka/Redpanda.

./gradlew :spring-messenger-example:bootRun

Run integration tests

Testcontainers starts PostgreSQL, RabbitMQ, and Redpanda to validate the full message path.

./gradlew :spring-messenger-example:test --tests '*ContainerTest'

Testing Helpers

The messenger-test module is separate from the example app. It gives reusable helpers for application tests: fake buses, recording event bus, event assertions, and handler invocation helpers. 

RecordingEventBus eventBus = new RecordingEventBus();
RegisterUserHandler handler = new RegisterUserHandler(repository, eventBus);

handler.handle(new RegisterUser("john@example.com", "John"));

assertThat(eventBus)
    .hasPublished(UserRegistered.class)
    .hasPublishedSatisfying(UserRegistered.class, event ->
        assertEquals("john@example.com", event.email())
    );
